Aspen uses a .aspen hidden directory in your website root as a catchall for files that don’t belong in your actual URL hierarchy. Think of it as your site’s junk drawer.

Previous versions of Aspen used a similar directory (cleverly named __) as a virtualenv with a UNIX-y userland layout. So conf files were under etc/, modules under lib/python/, and templates under etc/templates/. For reasons of versioning and deployment (most of which were first put into my head by Jim Fulton), it turns out to be better for .aspen to be flat, and any sandboxing to happen above and outside of your website root.

configuration files

Aspen will read aspen.conf and hooks.conf files from .aspen.

Python modules

The .aspen directory is placed on sys.path during startup, so you can place Python modules directly inside of this directory and they will be available to your hooks and simplates.

This is most helpful for pure-Python modules. C extension modules would need to be recompiled on different platforms. My preference there is to include the source code in .aspen, and build/install those into a short-lived virtualenv as part of deployment. You could also use the Python packaging tools, of course, either with the system Python or with a long-lived virtualenv.

Template bases and includes

The template page of simplates will look in .aspen for base templates for {% extends %} and file snippets for {% include %}. You can specify relative paths there, so if you would like to add a little structure to .aspen here you can.
